数据库设计

MySQL特点

  • 开源免费：其开源的特性使得开发者可以自由获取源代码，根据自身需求进行修改和定制，降低了使用成本，尤其受中小规模项目和创业公司青睐。
  • 跨平台支持：能在多种主流操作系统上运行，比如 Windows、Linux、macOS 等，方便不同操作系统环境下的用户部署和使用。
  • 性能出色：采用了多种优化机制来提高查询和数据处理的速度，像索引机制能快速定位数据，对于处理大量数据的场景也能有较好的响应性能，例如在电商网站处理海量商品信息和订单数据时表现良好。
  • 易于使用和管理：提供了简单直观的命令行界面（CLI）以及可视化的管理工具（如 phpMyAdmin 等），无论是有经验的数据库管理员还是新手开发者，都能较容易地进行数据库的创建、表结构设计、数据操作等工作。
  • 丰富的数据类型和函数支持：涵盖了整数、小数、字符串、日期时间等多种数据类型，还拥有大量内置函数用于数据处理、计算、字符串操作、日期处理等，满足各种复杂的业务逻辑需求。

user_info表

包含四个字段，分别为id、name、pwd、online

字段说明

  • id

类型为int

定义为主键(primary key)，意味着该字段的值在表中是唯一的，能够唯一标识每一条记录。

具备自增长的属性(auto_increament)，每次向表中插入一条新记录时，如果不手动指定id的值，数据库系统会自动为其分配一个一次递增的整数值，用于保证记录的唯一性和方便排序等操作。

  • name

类型为varchar(32)，表示它用来存储可变长度的字符串，最长可达 32 个字符。
有非空约束(not null)，意味着在插入或更新记录时，这个字段必须提供相应的值，不能为 NULL，用于确保每条用户记录都有对应的名字信息。

  • pwd

类型为 varchar(32) 类型，用于存储用户密码相关的字符串信息，最长 32 个字符。
也设置了 not null 约束，保证每条记录都必须有密码值，确保用户密码的完整性。不过在实际应用中，密码通常建议进行加密存储以提高安全性，这里只是简单的文本存储形式示例。

  • online

数据类型为 tinyint(1)，这是一种很小的整数类型，通常用于表示布尔值（在 MySQL 中，tinyint(1) 常用来模拟布尔值，0 表示假，1 表示真等情况）。
设置了默认值（default 0）为 0，也就是当插入一条新记录时，如果没有为该字段显式赋值，它会自动取值为 0，可用于表示用户初始状态是否在线，比如 0 表示离线，1 表示在线。

friend 表

用于存储用户之间的好友关系相关信息。
该表有四个字段，分别是 id、user_id、friend_id。

字段说明

  • id

    • 数据类型为 int，作为主键，具备唯一性，
    • 通过 auto_increment 实现自动增长，方便为每条好友关系记录赋予唯一标识。

  • user_id

    • int 类型。
    • 设置了not null约束，说明在插入记录时必须为该字段提供值，不能为空。
    • 有一个外键(foreign key)约束，关联到 user_info 表的 id 字段，这意味着 user_id 字段的值必须是user_info表中已经存在的 id 值，从而保证了好友关系中涉及的用户是在用户信息表中已经注册存在的真实用户，建立起了表与表之间的关联关系，用于维护数据的完整性和一致性。

  • friend_id

    • int 类型
    • 且 not null，要求插入记录时必须有值。
    • 同样设置了外键约束，关联到 user_info 表的 id 字段，确保所记录的好友的 id 也是在 user_info 表中存在的合法用户 id，与 user_id 字段一起，用来表示两个用户之间的好友关系（比如 user_id 为用户 A 的 id，friend_id 为用户 B 的 id，就表示用户 A 和用户 B 是好友关系）。
