线程池

发表于2024-12-04|更新于2024-12-22|学习笔记
|浏览量:

线程

​ 线程是进程中的一个执行任务（控制单元），是CPU分配和调度的基本单元，负责当前进程中程序的执行。

​ 如果把进程比喻成一个工厂，那么线程就是工厂里的一条生产线。一个进程可以包含多个线程，这些线程共享进程的资源，如内存空间、文件描述符等。

在一个文字处理软件（这是一个进程）中，有一个线程负责接收用户的键盘输入，另一个线程可能负责在屏幕上实时显示文字内容，还有一个线程可能负责自动保存文档。这些线程各自对应着不同的执行任务，它们在进程的环境下协同工作。

以一个网络浏览器进程为例，一个线程可能负责加载网页的 HTML 代码（这是一个执行路径），另一个线程可能负责解析和渲染页面中的图片（这是另一个执行路径），它们都在浏览器这个进程的大环境下运行，每个线程都有自己特定的任务序列，从开始任务到完成任务或者等待某些条件的过程构成了一个执行路径。

线程池原理

我们使用线程的时候就去创建一个线程，这样实现起来非常简便，但是如果并发的线程数量很多，并且每个线程都是执行一个时间很短的任务就结束了，这样频繁创建线程就会大大降低系统的效率，因为频繁创建线程和销毁线程需要时间。

那如果使用一种办法使得线程可以复用，就是执行完一个任务，并不被销毁，继续执行其他的任务。

线程池是一种多线程处理形式，处理过程中将任务添加到队列，然后在创建线程后自动启动这些任务。线程池线程都是后台线程。每个线程都使用默认的堆栈大小，以默认的优先级运行，并处于多线程单元中。如果某个线程在托管代码中空闲（如正在等待某个事件）,则线程池将插入另一个辅助线程来使所有处理器保持繁忙。如果所有线程池线程都始终保持繁忙，但队列中包含挂起的工作，则线程池将在一段时间后创建另一个辅助线程但线程的数目永远不会超过最大值。超过最大值的线程可以排队，但他们要等到其他线程完成后才启动。

线程池的组成

主要分为3个部分，这三部分配合工作就可以得到一个完整的线程池：

任务队列，存储需要处理的任务，由工作的线程来处理这些任务

  • 通过线程池提供的API函数，将一个待处理的任务添加到任务队列，或者从任务队列中删除
  • 已处理的任务会被从任务队列中删除
  • 线程池的使用者，也就是调用线程池函数往任务队列中添加任务的线程就是生产者线程

工作的线程（任务队列任务的消费者） ，N个

  • 线程池中维护了一定数量的工作线程, 他们的作用是是不停的读任务队列, 从里边取出任务并处理
  • 工作的线程相当于是任务队列的消费者角色
  • 如果任务队列为空, 工作的线程将会被阻塞 (使用条件变量/信号量阻塞)
  • 如果阻塞之后有了新的任务, 由生产者将阻塞解除, 工作线程开始工作

管理者线程（不处理任务队列中的任务），1个

  • 它的任务是周期性的对任务队列中的任务数量以及处于忙状态的工作线程个数进行检测
  • 当任务过多的时候, 可以适当的创建一些新的工作线程
  • 当任务过少的时候, 可以适当的销毁一些工作的线程

线程池优缺点

优点

提高性能和资源利用率

线程的创建和销毁是有成本的。当频繁地创建和销毁线程时，会消耗大量的系统资源，包括 CPU 时间和内存。

创建一个线程需要为其分配栈空间，在 Linux 系统中，默认的线程栈大小可能是几兆字节（如 8MB 左右）。线程池中的线程是预先创建好的，当有任务到来时，可以直接使用已有的线程，避免了频繁创建和销毁线程的开销。

假设一个简单的 Web 服务器，它需要处理大量的 HTTP 请求。如果没有线程池，每次收到一个请求就创建一个新线程来处理，处理完后销毁线程。当请求频繁时，系统会花费大量时间在创建和销毁线程上，而不是真正处理请求。而使用线程池，预先创建好一定数量的线程等待请求，能显著提高服务器对请求的处理效率。

控制并发线程数量

可以通过配置线程池的大小来限制同时执行的线程数量。这有助于防止系统因过多的线程同时运行而导致资源耗尽，造成内存溢出或 CPU 过度占用。

在一个数据库连接池中，设置合适的线程池大小可以避免同时建立过多的数据库连接，防止数据库服务器因连接数过多而崩溃。如果数据库服务器最多支持 100 个并发连接，那么线程池大小可以根据实际情况设置为一个合理的值，如 80，这样就可以有效地控制对数据库的并发访问。

便于管理和维护线程生命周期

线程池统一管理线程的生命周期。它可以对线程进行状态监控，知道哪些线程是空闲的，哪些线程正在执行任务。可以根据任务的优先级、执行时间等因素来调度线程。

在一个复杂的企业级应用中，有多种不同类型的任务需要执行，如数据备份任务、用户请求处理任务等。通过线程池，可以根据任务的性质和优先级，将任务分配到合适的线程中，并且可以方便地对线程进行重启、暂停等操作，以应对不同的应用场景。

提高响应速度

由于线程池中的线程是预先创建的，当任务到达时可以立即分配线程进行处理，减少了任务等待线程创建的时间，从而提高了系统对任务的响应速度。

在一个图形处理软件中，用户频繁地进行图像缩放、旋转等操作。使用线程池，当用户发出操作指令后，线程池中的线程可以马上开始处理图像，用户可以更快地看到操作结果，提升了用户体验。

缺点

增加系统复杂性

线程池的实现和管理本身比较复杂。需要考虑线程池的大小如何设置才合理，如何处理线程的空闲和繁忙状态，如何处理任务队列等诸多问题。如果线程池设计不当，可能会导致性能下降或者出现死锁等问题。

在一个高并发的分布式系统中，如果线程池的任务队列没有合理的容量限制，当大量任务涌入时，可能会导致内存溢出。或者如果线程池的调度策略不合理，可能会导致某些重要任务长时间得不到执行。

调试困难

当多个线程在一个线程池中执行任务时，可能会出现并发问题，如数据竞争、死锁等。这些问题很难调试，因为它们可能是由多个线程之间复杂的交互和执行顺序引起的。

可能存在资源浪费

如果线程池的大小设置不合理，可能会导致资源浪费。如果线程池设置得过大，会占用过多的内存资源，因为每个线程都需要一定的栈空间等资源。而如果线程池中的任务很少，大量线程会处于空闲状态，也会浪费 CPU 资源用于线程的维护。

在一个小型的本地应用程序中，设置了一个拥有 100 个线程的线程池，但实际应用场景中很少会有超过 10 个任务同时执行，那么这就造成了大量线程资源的浪费
